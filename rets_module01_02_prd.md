## 1. 제품 개요

### 1.1 한 줄 요약

> **RETS-MODULE-01**은 초안 수준의 RFP 요구사항을 LLM을 활용하여 구체적인 개발요구사항으로 자동 상세화하고, 3개 차원의 품질 진단을 통해 요구공학 정제(refinement) 업무를 체계적으로 지원하는 **웹 기반 지능형 요구사항 관리 도구**이다.

### 1.2 핵심 가치 제안

| 가치 | 설명 |
|------|------|
| **자동화** | LLM이 모호한 RFP 요구사항을 구체적인 DEV-REQ로 자동 분해·정제하여 분석 공수를 절감 |
| **품질 가시화** | 명확성·일관성·검증가능성 3개 차원의 정량 진단 결과를 하이라이팅으로 즉각 시각화 |
| **추적성 확보** | RFP-REQ ↔ DEV-REQ 간 양방향 추적 관계를 자동 생성하여 요구사항 근거 관리 체계화 |
| **협업 지원** | 진단 결과를 Markdown/CSV로 내보내어 리뷰·승인 프로세스에 즉시 활용 가능 |

## 2. 배경 및 문제 정의

### 2.1 현황 및 문제

현재 당사의 요구공학 업무는 다음과 같은 구조적 문제를 안고 있다.

**문제 1. 요구사항 상세화 업무의 수작업 의존성**

RFP에서 수집된 요구사항은 대부분 추상적·개괄적 수준으로 기술되어 있으며, 이를 구현 가능한 개발요구사항 수준으로 분해·정제하는 작업이 담당자의 경험과 역량에 전적으로 의존하고 있다. 이로 인해 팀·담당자마다 산출물의 품질 편차가 크게 발생하고, 상당한 반복 작업이 수반된다.

**문제 2. 요구사항 품질의 정성적 의존성**

요구사항의 모호성(ambiguity), 비일관성(inconsistency), 검증 불가능성(unverifiability) 등 품질 이슈가 초기에 발견되지 않고, 설계·구현 단계에 이르러서야 드러나는 경우가 빈번하다. 이는 재작업(rework) 비용을 크게 증가시키는 요인으로 작용한다.

**문제 3. 요구사항 데이터의 파편화 및 추적 불가**

수집된 RFP 요구사항과 이를 기반으로 도출된 개발요구사항 사이의 연결 관계가 체계적으로 관리되지 않아, 변경 발생 시 영향 범위 파악이 어렵다.

### 2.2 솔루션 방향

RETS-MODULE-01은 이러한 문제를 해소하기 위해:

1. **LLM 기반 지능형 보조**로 상세화 초안을 자동 생성하고, 사용자가 이를 검토·편집하는 Human-in-the-Loop 방식을 채택
2. **정량적 품질 진단 엔진**으로 요구사항 품질 이슈를 명세 단계에서 조기 식별
3. **구조화된 JSON 스키마** 기반 데이터 관리로 타 모듈과의 연계 및 추적성 보장

- **사업 발주사(고객사)**: PoC 평가 및 모듈 검수

## 5. 유저 스토리 (RFP 기반 상세화)

RFP 2.3.1절의 핵심 유저 스토리를 다음과 같이 상세화한다.

---

### US-01. RFP-REQ 파일 불러오기 및 그리드 출력

**As a** BA(요구공학 분석가),  
**I want to** JSON 형식의 RFP 요구사항 파일을 시스템에 불러와 목록으로 확인하고 싶다,  
**So that** 별도의 수작업 입력 없이 기존 RFP 데이터를 즉시 분석 작업에 활용할 수 있다.

**수락 기준 (Acceptance Criteria)**:

- [ ] [불러오기] 버튼 클릭 시 파일 선택 다이얼로그가 열린다.
- [ ] 선택한 JSON 파일이 `rets_02c_req_schema.json` 스키마를 준수할 경우, 요구사항 목록 그리드에 파싱된 데이터가 정상 출력된다.
- [ ] 스키마 오류가 있는 파일 선택 시 오류 안내 메시지가 표시된다.
- [ ] 로드된 요구사항의 총 건수가 그리드 상단에 표시된다.
- [ ] `req_type`, `req_id`, `req_name`, `description`, `functional_class`, `priority` 필드가 기본 컬럼으로 그리드에 표시된다.
- [ ] 계층(Level)에 따라 들여쓰기 또는 트리 구조로 요구사항이 시각화된다.

---

### US-02. 요구사항 CRUD (RFP-REQ / DEV-REQ 공통)

**As a** BA,  
**I want to** 요구사항 목록에서 항목을 직접 추가·수정·삭제할 수 있기를 원한다,  
**So that** LLM 제안과 무관하게 수동으로 요구사항을 관리하고 최신 상태를 유지할 수 있다.

**수락 기준**:

- [ ] 그리드에서 직접 셀 편집(인라인 수정) 또는 상세 편집 패널을 통해 필드 수정이 가능하다.
- [ ] [+ 추가] 버튼으로 신규 요구사항 행을 추가할 수 있다. 추가 시 `req_id` 자동 채번 제안.
- [ ] 행 선택 후 [삭제] 버튼으로 요구사항을 삭제할 수 있다. 삭제 전 확인 다이얼로그가 표시된다.
- [ ] 요구사항의 `req_level`(1~5)을 설정하여 계층 구조를 구성할 수 있다.
- [ ] 수정 내용은 즉시 메모리 상에 반영되며, [저장] 버튼 또는 자동 저장으로 유지된다.
- [ ] 변경 이력(`change_history`)이 자동 기록된다.

---

### US-03. DEV-REQ JSON 내보내기

**As a** BA,  
**I want to** 편집 완료된 개발요구사항 목록을 JSON 파일로 내보내고 싶다,  
**So that** 타 모듈(MODULE-03, 04, 05)에서 DEV-REQ를 입력 데이터로 활용할 수 있다.

**수락 기준**:

- [ ] 그리드 상단 [DEV-REQ 내보내기] 버튼 클릭 시 `rets_02c_req_schema.json` 스키마를 준수하는 JSON 파일이 다운로드된다.
- [ ] 내보내기 대상은 현재 `req_type = DEV`인 항목 전체이다.
- [ ] 파일명은 `rets_module01_dev_req_export_YYYYMMDD.json` 형식으로 자동 생성된다.

---

### US-04. LLM 기반 RFP-REQ 자동 상세화 (DEV-REQ 명세)

**As a** BA,  
**I want to** 선택한 RFP-REQ에 대해 LLM이 구체적인 DEV-REQ 목록을 자동으로 제안받고 싶다,  
**So that** 상세화 초안 작성 시간을 단축하고, 누락 없는 요구사항 분해를 지원받을 수 있다.

**수락 기준**:

- [ ] 그리드에서 1개 이상의 RFP-REQ를 선택하고 [AI 상세화] 버튼을 클릭하면 LLM 호출이 시작된다.
- [ ] LLM 호출 중 로딩 인디케이터(스피너 또는 진행 표시)가 표시된다.
- [ ] LLM 응답을 파싱하여 제안된 DEV-REQ 목록이 별도 패널(사이드 패널 또는 모달)에 표시된다.
- [ ] 제안된 DEV-REQ는 항목 단위로 [수락] / [거부] / [편집 후 수락]이 가능하다.
- [ ] 수락된 DEV-REQ는 그리드에 자동 추가되며, `source_rfp_ids` 필드에 원본 RFP-REQ ID가 자동 매핑된다.
- [ ] 사용자가 수동으로 DEV-REQ를 추가하거나 LLM 제안 항목을 직접 수정할 수 있다.
- [ ] LLM 호출 실패 시 오류 메시지와 재시도 옵션이 제공된다.

---

### US-05. 명확성 진단 (Clarity Diagnosis)

**As a** BA,  
**I want to** DEV-REQ 목록에서 모호한 표현이 포함된 항목을 자동으로 식별하고 싶다,  
**So that** 명세 단계에서 모호성을 제거하고 명확한 요구사항을 확보할 수 있다.

**수락 기준**:

- [ ] [명확성 진단] 버튼 클릭 시 LLM이 모든 DEV-REQ의 모호성을 분석한다.
- [ ] 모호성이 탐지된 요구사항 행이 시각적으로 하이라이팅된다 (예: 노란색 배경 또는 경고 아이콘).
- [ ] 각 하이라이팅 항목에 마우스 오버 또는 클릭 시 구체적인 모호 표현과 개선 제안이 툴팁/패널로 표시된다.
- [ ] 진단 결과는 항목별로 `명확성 점수(0~5)` 및 이슈 설명을 포함한다.

---

### US-06. 일관성 진단 (Consistency Diagnosis)

**As a** BA,  
**I want to** DEV-REQ 목록 내 서로 충돌하거나 비일관적인 요구사항 쌍을 자동으로 식별하고 싶다,  
**So that** 이해관계자 간 해석 충돌을 사전에 방지할 수 있다.

**수락 기준**:

- [ ] [일관성 진단] 버튼 클릭 시 LLM이 전체 DEV-REQ를 대상으로 비일관성(충돌) 분석을 수행한다.
- [ ] 충돌이 감지된 요구사항 쌍이 하이라이팅된다 (예: 빨간색 테두리).
- [ ] 충돌하는 요구사항 쌍과 충돌 원인이 패널에 상세히 표시된다.
- [ ] 진단 결과에는 충돌 유형(용어 불일치, 기능 상충, 중복 명세 등)이 분류되어 표시된다.

---

### US-07. 검증가능성 진단 (Verifiability Diagnosis)

**As a** BA,  
**I want to** DEV-REQ 항목별로 테스트 또는 검증 가능 여부에 대한 피드백을 받고 싶다,  
**So that** QA 단계에서 검증 불가한 요구사항으로 인한 문제를 사전에 방지할 수 있다.

**수락 기준**:

- [ ] [검증가능성 진단] 버튼 클릭 시 LLM이 각 DEV-REQ의 검증 가능성을 평가한다.
- [ ] 검증 불가 또는 개선이 필요한 항목이 하이라이팅된다.
- [ ] 각 항목에 검증가능성 평가 코멘트 및 개선 제안이 표시된다.
- [ ] 검증가능성 점수(0~5)가 항목별로 표시된다.

---

### US-08. 품질 진단 보고서 생성 및 출력

**As a** PM,

## 10. 기술 고려사항

### 10.1 LLM 프롬프트 엔지니어링 전략

LLM 기반 기능의 품질은 프롬프트 설계에 크게 의존하므로, 다음 전략을 적용한다.

| 기능 | 프롬프트 전략 |
|------|-------------|
| **RFP→DEV 상세화** | 시스템 프롬프트에 요구공학 전문가 역할 부여. 입력된 RFP-REQ의 `req_name`, `description`을 컨텍스트로 제공. 출력 형식: `JSON 배열 (DEV-REQ 스키마 준수)` 명시 |
| **명확성 진단** | SMART 기준(Specific, Measurable, Achievable, Relevant, Time-bound) 기반 모호성 판단 기준을 시스템 프롬프트에 포함. 출력 형식: `[{req_id, clarity_score, issues: []}]` JSON |
| **일관성 진단** | 전체 DEV-REQ 목록을 한 번에 제공하여 교차 비교 수행. 출력 형식: `[{req_id_a, req_id_b, conflict_type, description}]` JSON |
| **검증가능성 진단** | ISO/IEC 29148 기반 검증가능성 기준 명시. 출력 형식: `[{req_id, verifiability_score, feedback}]` JSON |

모든 LLM 호출에서 응답은 **순수 JSON 형식**으로만 반환하도록 강제하며, 코드 블록(`\`\`\`json`) 래핑을 포함하는 경우를 대비한 파싱 로직을 구현한다.

### 10.2 데이터 영속성 전략

백엔드 없이 브라우저 환경에서 동작하므로, 다음 방식으로 데이터 영속성을 확보한다:

| 데이터 유형 | 저장 방식 |
|-------------|-----------|
| 요구사항 목록 (작업 중 세션) | 메모리(JavaScript 변수) + `sessionStorage` |
| 사용자 설정 (team_id, 언어) | `localStorage` |
| 작업 내용 자동 저장 | `localStorage` (JSON 직렬화, 5분 간격 또는 변경 시) |

### 10.3 비동기 처리 원칙

RFP 2.1.2.2의 요구에 따라, 1초 이상 소요될 수 있는 모든 작업은 비동기 방식으로 처리한다:

- LLM API 호출: `async/await` + `AbortController` (타임아웃 30초)
- 파일 읽기/쓰기: `FileReader API` (비동기)
- 대용량 JSON 파싱: `setTimeout` 기반 청크 처리 (필요 시)

### 10.4 LLM 릴레이 서버 연동

```
[클라이언트 (HTML)] ──HTTPS──▶ [LLM 릴레이 서버]
                                  https://ai-relay.mooo.com:11443
                                        │
                                        ▼
                               [Google Gemini API]

호출 흐름:
1. POST /chat/session/create  { team_id, system_prompt }
   → session_id 반환
2. POST /chat/session/{session_id}/message  { message }
   → { response: "...JSON..." }
3. 클라이언트에서 response 내 JSON 파싱
```
