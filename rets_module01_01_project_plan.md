## 1. 프로젝트 개요

### 1.1 사업 배경

본 프로젝트는 「사내 요구공학 도구 스위트(RETS; Requirements Engineering Tool Suite)」 구축 사업의 일환으로, 당사 내 요구공학 업무의 체계화·표준화·고도화를 목적으로 한다. 현재 요구사항 명세·상세화·정제 업무의 대부분이 수작업으로 이루어지고 있으며, 요구공학 활동의 품질이 담당자 개인의 역량에 크게 의존하고 있는 한계를 해소하기 위해 본 도구를 개발한다.

### 1.2 모듈 개요

**RETS-MODULE-01**은 초안 수준의 개괄적이고 모호한 RFP 요구사항(RFP-REQ)을 입력받아, LLM 기반 지능을 활용하여 이를 보다 구체적이고 일관된 개발요구사항(DEV-REQ) 형태로 상세화하고, 품질 기준에 따라 정량적으로 진단·평가할 수 있는 기능과 화면을 제공하는 웹 기반 도구이다.

본 모듈은 요구사항 문서의 모호성(ambiguity), 누락(incompleteness), 중복(redundancy), 비일관성(inconsistency)을 조기에 식별하여 정제(refinement) 업무를 체계적으로 지원한다.

### 1.3 개발 방식

- **구현 형태**: Vanilla JS 기반 단일 HTML 파일 (`rets_module01_07_app.html`)
- **LLM 연동**: LLM 릴레이 서버(Google Gemini) 활용
- **권장 방법론**: 테스트 주도 개발(TDD; Test-Driven Development)
- **백엔드**: 없음 (프론트엔드 전용 PoC)

| **UX 목표** | 요구공학 담당자가 직관적으로 사용 가능한 그리드+편집 인터페이스 제공 |

### 2.2 성공 기준 (DoD; Definition of Done)

- [ ] RFP 제공 벤치마크 JSON 파일을 오류 없이 파싱하여 그리드에 출력
- [ ] LLM을 통해 RFP-REQ 1건 이상을 DEV-REQ로 자동 상세화 및 편집 완료
- [ ] 명확성·일관성·검증가능성 3종 품질 진단 버튼 정상 동작
- [ ] 품질 진단 보고서 Markdown/CSV 형식으로 파일 출력 성공
- [ ] DEV-REQ 목록 JSON 내보내기 기능 정상 동작
- [ ] 공통 디자인 요구사항 충족 (RETS 디자인 시스템 준수)
- [ ] 한글/영어 다국어 전환 기능 동작
- [ ] SRS 기능 요구사항 커버리지 100% 충족
- [ ] PoC 발표 세션(5분) 성공적 수행

## 4. 개발 범위 요약

### 4.1 In-Scope (개발 대상)

| 기능 영역 | 세부 내용 |
|-----------|-----------|
| **요구사항 관리 (CRUD)** | RFP-REQ 및 DEV-REQ에 대한 생성·조회·수정·삭제, 계층적 구조(Level 1~5) 지원 |
| **파일 I/O** | JSON 형식 RFP-REQ 불러오기, DEV-REQ JSON 내보내기, 품질 진단 보고서 Markdown/CSV 내보내기 |
| **LLM 기반 자동 상세화** | RFP-REQ → DEV-REQ 자동 상세화(분해·정제) 제안, 사용자 수락/거부/편집 |
| **추적성 관리** | RFP-REQ ↔ DEV-REQ 간 트레이서빌리티 생성 및 표시 |
| **품질 진단** | 명확성(Clarity), 일관성(Consistency), 검증가능성(Verifiability) 3종 LLM 기반 진단 |
| **품질 보고서 출력** | 진단 결과를 Markdown 또는 CSV 파일로 출력 |
| **대시보드** | 요구사항 현황(RFP-REQ/DEV-REQ 건수, 품질 점수 요약) 표시 |
| **다국어 지원** | 한글/영어 UI 전환 |

### 4.2 Out-of-Scope (개발 제외)

- 백엔드 서버 또는 데이터베이스 구축
- 타 모듈(MODULE-02~05)과의 실시간 연동
- 사용자 인증/권한 관리
- 모바일 최적화 (데스크톱 웹 기준 구현)

### 9.2 대응 계획 상세

**R-01 / R-02 (LLM 연동 관련)**

LLM 호출이 요구되는 모든 기능에 대해 사용자 경험을 저해하지 않도록 로딩 인디케이터를 제공하고, 실패 시 명확한 오류 안내와 재시도 버튼을 UI에 구현한다. 시스템 프롬프트 내 `JSON 형식으로만 응답할 것`을 명시하여 파싱 안정성을 높인다.

**R-04 (일정 지연)**

전체 기능을 다음과 같이 우선순위로 분류한다:

| 우선순위 | 기능 |
|----------|------|
| **P1 (Must-have)** | JSON 불러오기·내보내기, 요구사항 CRUD, LLM 상세화(1건 이상), 품질 진단 3종 |
| **P2 (Should-have)** | 계층 구조(Level 1~5), 추적 매트릭스, 품질 보고서 CSV 출력 |
| **P3 (Nice-to-have)** | 대시보드 시각화, 다국어 전환, 자동 저장 |

P1을 STEP 3 중반까지 완료하는 것을 목표로 하며, 일정 압박 발생 시 P3를 후순위로 조정한다.

**R-06 (CORS 차단)**

HTML 파일을 직접 브라우저에서 열 경우 발생할 수 있는 CORS 문제를 사전에 파악하고, GitHub Pages 배포 환경에서 릴레이 서버 호출 허용 여부를 개발 착수 전에 확인한다.
